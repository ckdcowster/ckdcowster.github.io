REST Best Practices

When designing and working with RESTful APIs, following best practices ensures your endpoints are intuitive, scalable, secure, and maintainable. Here are some key principles and good practices:

  1. Use Nouns for Resource URIs
    • URIs should represent resources, not actions
    • Good: GET /users/123
    • Bad: GET /getUserById/123
  2. Use HTTP Methods Appropriately
    • GET – Retrieve data
    • POST – Create a new resource
    • PUT – Update an existing resource (replace)
    • PATCH – Partially update a resource
    • DELETE – Remove a resource
  3. Use Plural Nouns
    • Keep naming consistent and plural for collections
    • Example: /users, /products, /orders
  4. Use Status Codes Meaningfully
    • 200 OK – Successful GET, PUT, or DELETE
    • 201 Created – Successful POST
    • 204 No Content – Successful DELETE with no return
    • 400 Bad Request – Invalid input
    • 401 Unauthorized – Authentication required
    • 403 Forbidden – Authenticated but not allowed
    • 404 Not Found – Resource doesn’t exist
    • 429 Too Many Requests – Client error
    • 500 Internal Server Error – Server-side issue
    • 300 Moved Permanently – Redirection status
  5. Version Your API
    • Use URI versioning: /api/v1/users
    • Helps manage breaking changes over time
  6. Use Query Parameters for Filtering, Sorting, and Pagination
    • Example: /products?category=books&sort=price_asc&page=2&limit=20
  7. Secure Your API
    • Use HTTPS
    • Implement authentication (e.g., JWT, OAuth2)
    • Validate and sanitize inputs to prevent injection attacks
    • Rate limit to prevent abuse
  8. Request & Response Format
    • Use JSON as the standard format
    • Include metadata (e.g., pagination info) in responses
    • Use consistent field naming (e.g., camelCase or snake_case, but be consistent)
  9. Documentation and Testing
    • Use tools like Swagger/OpenAPI for documentation
    • Provide examples and test endpoints
    • Ensure your API is covered with automated tests

Examples

HTTP Method API Result
GET /api/v1/admin/software/isos List all software ISO Images and their details
POST /api/v1/admin/software/isos Create a new ISO image
GET /api/v1/admin/software/isos/{id} Get details of a specific ISO image
PUT /api/v1/admin/software/isos/{id} Update details of a specific ISO image
DELETE /api/v1/admin/software/isos/{id} Deletes a specific ISO image
GET /api/v1/admin/software/isos/downloads List all download histories for all ISO images*
GET /api/v1/admin/software/isos/{id}/downloads List specific ISO download history**
GET /api/v1/software/isos/{id}/file Download a specific ISO image**

* So while /isos/downloads isn't a textbook sub-resource (because it's not tied to a specific ISO), it's a reasonable compromise if:

  • You want to make the scope obvious
  • You don't support downloads for other resource types
  • You're not exposing a global /downloads endpoint
If you want to keep RESTful structure and clarity, consider: GET /downloads?resource=iso

** This structure is more idiomatic in REST because it treats file / downloads as a sub-resource of the ISO identified by {id}.
You're essentially saying: "Give me the file associated with this specific ISO."

What is an Idempotent API?

An idempotent HTTP method is a method that can be invoked many times without different outcomes.
It should not matter if the method has been called only once, or ten times over. The result should always be the same.

Idempotency with HTTP Methods

If we follow the REST principles in designing our APIs, we will have automatically idempotent REST APIs for: GET, PUT, DELETE, HEAD, OPTIONS, and TRACE methods.
Only POST and PATCH APIs will not be idempotent.

  • POST and PATCH are NOT idempotent
  • GET, PUT, DELETE, HEAD, OPTIONS and TRACE are idempotent
HTTP Method Idempotent Safe
GET ✅ ✅
HEAD ✅ ✅
PUT ✅ ❌
DELETE ✅ ❌
POST ❌ ❌
PATCH ❌ ❌
